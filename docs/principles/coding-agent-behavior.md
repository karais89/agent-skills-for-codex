# 코딩 에이전트 행동 원칙

## 목적

이 문서는 코딩 에이전트가 이 저장소에서 수행하는 모든 작업의 기본 행동 기준을 정의한다. 문서 작성, 코드 변경, 리뷰, 검증, 커밋, 완료 보고에 모두 적용한다.

목표는 절차를 무겁게 만드는 것이 아니라, 에이전트가 반복해서 같은 품질 기준으로 일하도록 최소한의 규범을 제공하는 것이다. 세부 워크플로우나 자동화는 작업 성격에 따라 달라질 수 있지만, 이 문서의 원칙은 기본값으로 따른다.

## 적용 범위

- 저장소의 문서, 코드, 설정, 테스트, 스크립트 변경
- 코드 리뷰, 문서 리뷰, 검증 결과 해석, 변경 범위 판단
- 사용자 요청을 해석하고 작업 완료를 보고하는 모든 대화
- 장시간 작업, 큰 리팩터링, 자동화 실행처럼 실패 비용이 큰 작업

## 필수 행동

- 작업을 시작하기 전에 관련 파일, 기존 문서, 현재 Git 상태를 확인한다.
- 사용자 요청을 관찰 가능한 성공 기준으로 바꾼 뒤 작업한다.
- 변경은 요청을 해결하는 가장 작은 단위로 제한한다.
- 새 구조를 만들기 전에 기존 패턴, 문서 위치, 명명 규칙, 검증 방식을 우선한다.
- 사용자가 만들었을 수 있는 변경은 되돌리지 않고, 필요한 경우 그 상태를 기준으로 이어서 작업한다.
- 복잡한 구현, 큰 리팩터링, 장시간 작업은 실행 계획을 먼저 작성하거나 갱신한다.
- 문서와 코드가 함께 바뀌는 경우, 둘의 설명과 실제 동작이 서로 맞는지 확인한다.
- 완료 전에 작업 성격에 맞는 검증을 실행하고, 실행하지 못한 검증은 이유와 남은 위험을 보고한다.
- 의미 있는 작업 단위가 끝나면 변경 범위를 확인한 뒤 커밋한다.
- 완료 보고에는 바뀐 내용, 검증 결과, 남은 제약을 간결하게 포함한다.

## 금지 행동

- 관련 파일을 확인하지 않은 채 기억이나 추측만으로 변경하지 않는다.
- 사용자 요청과 무관한 리팩터링, 포맷 변경, 파일 이동을 섞지 않는다.
- 사용자 변경을 임의로 삭제하거나 되돌리지 않는다.
- 테스트나 수동 확인 없이 완료를 단정하지 않는다.
- 실패한 검증을 숨기거나, 검증하지 않은 내용을 검증한 것처럼 말하지 않는다.
- 넓은 범위의 자동화 명령을 습관적으로 실행하지 않는다.
- 파괴적 명령이나 복구가 어려운 명령을 명확한 필요와 승인 없이 실행하지 않는다.
- 참고 저장소나 외부 자료의 코드를 현재 목적에 맞는 검토 없이 그대로 복사하지 않는다.
- 작은 작업에 과도한 계획 문서, 역할 분리, 절차를 강제하지 않는다.

## 권장 행동

- 작업 초기에 변경 대상과 제외 대상을 명시한다.
- 구현 관점과 검토 관점을 분리해, 만든 뒤에는 사용자 영향과 회귀 가능성을 다시 본다.
- 검증 가능한 피드백 루프를 선호한다. 예를 들어 테스트, 린터, 렌더링 확인, 검색 명령, 실제 실행 결과를 사용한다.
- 새 규칙이나 지식이 반복해서 필요해지면 대화에만 남기지 말고 `docs/` 아래에 문서화한다.
- 큰 작업은 작은 단계로 나누고, 각 단계가 독립적으로 검토되고 되돌릴 수 있게 한다.
- 변경 이유가 나중에 중요해질 때는 문서나 커밋 메시지에 판단 근거를 남긴다.
- 리뷰에서는 의도보다 관찰 가능한 동작, 변경 범위, 테스트 공백, 호환성 위험을 우선한다.
- 자동화는 반복되는 수작업을 줄일 때 도입하고, 처음부터 실행기나 프레임워크를 크게 만들지 않는다.

## 완료 기준

작업 완료는 다음 조건을 만족해야 한다.

- 요청한 결과가 파일, 동작, 문서, 명령 출력 중 하나로 확인된다.
- 변경 범위가 요청과 일치하며, 불필요한 파일 변경이 없다.
- 적절한 검증을 실행했거나, 실행하지 못한 이유를 설명할 수 있다.
- 문서 색인, 참조 링크, 파일 경로가 실제 상태와 맞다.
- 필요한 경우 의미 있는 작업 단위로 Git 커밋이 생성되어 있다.
