# OpenAI Codex 실행 계획 참고

원문: https://developers.openai.com/cookbook/articles/codex_exec_plans

확인일: 2026-04-26

## 핵심 요약

OpenAI Cookbook의 실행 계획 문서는 여러 시간 이상 걸리는 복잡한 구현이나 큰 리팩터링을 Codex가 안정적으로 수행하도록 `PLANS.md` 같은 살아 있는 계획 문서를 사용하는 방식을 설명한다. 계획 문서는 작업 전 설계를 검토하게 해주고, 작업 중에는 진행 상황과 결정 사항을 계속 갱신하는 기준점이 된다.

## 사용할 때

- 기능 범위가 크거나 여러 파일과 모듈을 동시에 바꾸는 경우
- 리팩터링, 마이그레이션, 아키텍처 변경처럼 되돌리기 어렵거나 검증 부담이 큰 경우
- 중간에 작업을 멈췄다가 다시 이어갈 가능성이 높은 경우
- 사람이나 다른 에이전트가 계획을 읽고 같은 결과를 재현해야 하는 경우

## 좋은 실행 계획의 구성

- 목적: 변경 후 사용자가 얻는 결과와 확인 방법을 설명한다.
- 진행 상황: 체크박스와 시간 기록으로 현재 상태를 계속 갱신한다.
- 발견 사항: 구현 중 알게 된 예상 밖의 동작, 버그, 제약을 근거와 함께 남긴다.
- 결정 로그: 중요한 선택과 이유, 날짜와 작성자를 기록한다.
- 결과 회고: 완료 시 결과, 남은 간극, 배운 점을 정리한다.
- 맥락: 관련 파일, 모듈, 용어를 처음 읽는 사람도 이해할 수 있게 설명한다.
- 작업 계획: 어떤 파일의 어느 부분을 어떻게 바꿀지 구체적으로 쓴다.
- 실행 단계: 실행할 명령과 기대 결과를 적는다.
- 검증 기준: 테스트나 수동 확인 방법을 관찰 가능한 동작으로 정의한다.
- 복구 방법: 반복 실행 가능 여부와 실패 시 회복 절차를 적는다.
- 산출물: 중요한 로그, diff, 예시를 필요한 만큼만 남긴다.
- 인터페이스와 의존성: 최종적으로 존재해야 하는 타입, 함수, 모듈, 외부 의존성을 명확히 한다.

## 작성 원칙

- 계획은 단순한 할 일 목록이 아니라 독립적으로 실행 가능한 설계 문서여야 한다.
- 작업이 진행되면 계획 문서도 함께 갱신한다.
- 무엇을 할지뿐 아니라 왜 그렇게 하는지도 기록한다.
- 계획을 수정할 때는 관련 섹션 전체가 새 상태와 일치하는지 확인한다.
- 작은 변경에는 과도한 계획 문서를 만들지 않는다.
