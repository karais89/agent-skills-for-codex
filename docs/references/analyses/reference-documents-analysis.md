## 전체 결론

세 문서는 같은 문제를 다른 층위에서 다룬다. OpenAI Harness Engineering은 저장소 전체를 에이전트가 일하기 좋은 환경으로 만드는 운영 철학을 제시한다. OpenAI Codex 실행 계획 문서는 개별 큰 작업을 어떻게 문서화하고 끝까지 유지할지 다룬다. Anthropic 하네스 설계 문서는 장시간 자율 작업에서 역할을 어떻게 나누고 평가 루프를 어떻게 설계할지 설명한다.

이 프로젝트에는 세 문서를 다음처럼 조합하는 것이 적합하다.

- 저장소 운영 철학은 OpenAI Harness Engineering을 따른다.
- 큰 작업의 계획 문서 형식은 OpenAI Codex 실행 계획을 따른다.
- 장시간 구현과 UI 검증 루프는 Anthropic 하네스 설계의 역할 분리 방식을 따른다.

## 공통 원칙

### 1. 에이전트가 볼 수 없는 지식은 운영 지식이 아니다

세 문서 모두 중요한 정보가 대화, 사람의 기억, 외부 문서에만 있으면 장시간 작업 품질이 떨어진다고 본다. 저장소 안의 문서, 계획, 상태 파일, 테스트, 로그처럼 에이전트가 직접 읽을 수 있는 형태로 남겨야 한다.

이 프로젝트의 적용:

- `AGENTS.md`는 짧은 진입점으로 유지한다.
- 실제 지식은 `docs/`에 분리해 저장한다.
- 작업 중 결정, 발견, 검증 결과는 대화가 아니라 문서에 남긴다.

### 2. 계획은 일회성 안내가 아니라 작업 상태다

OpenAI Exec Plans는 계획 문서를 살아 있는 문서로 취급한다. Anthropic 문서의 스프린트 계약과 파일 기반 핸드오프도 같은 방향이다. 계획은 “처음에 쓴 설계”가 아니라 현재 진행 상황, 발견 사항, 결정, 남은 일을 반영하는 작업 상태여야 한다.

이 프로젝트의 적용:

- 큰 작업은 `docs/plans/active/` 아래에 계획 문서로 관리한다.
- 완료된 계획은 `docs/plans/completed/`로 옮긴다.
- 계획 문서에는 진행 상황, 결정 로그, 검증 기준, 복구 방법을 포함한다.

### 3. 검증은 에이전트 바깥에 둔다

세 문서 모두 자기 평가만으로는 부족하다는 방향을 공유한다. OpenAI Harness는 테스트·린터·관측 가능성 같은 피드백 루프를 강조하고, Anthropic 문서는 구현자와 평가자를 분리한다. 실행 계획 문서는 검증 기준을 구체적인 명령과 관찰 가능한 동작으로 적으라고 요구한다.

이 프로젝트의 적용:

- 완료 선언 전에 실행 가능한 검증 명령을 우선 확인한다.
- UI 작업은 브라우저 조작, 스크린샷, 사용자 흐름 검증을 포함한다.
- 중요한 작업은 구현 리뷰와 검증 리뷰를 분리한다.

### 4. 큰 작업은 작게 쪼개되 맥락은 잃지 않는다

작업을 작게 쪼개면 각 단위의 성공 기준이 선명해진다. 다만 쪼갠 단위 사이의 맥락이 사라지면 다음 단계가 흔들린다. 그래서 실행 계획, 스프린트 계약, step summary, 결정 로그 같은 연결 장치가 필요하다.

이 프로젝트의 적용:

- 작업 단위는 독립적으로 검증 가능해야 한다.
- 각 단위가 끝나면 요약과 다음 단위에 필요한 맥락을 남긴다.
- 단순한 작은 변경에는 과한 계획 문서를 만들지 않는다.

### 5. 하네스는 모델의 약점을 보완하는 장치다

Anthropic 문서는 모델 성능이 변하면 하네스 구성 요소도 달라져야 한다고 설명한다. OpenAI Harness도 도구, 문서, 피드백 루프를 에이전트가 안정적으로 일하게 만드는 환경으로 본다. 따라서 하네스는 고정된 의식 절차가 아니라 현재 모델과 작업 난이도에 맞춰 조정하는 운영 장치다.

이 프로젝트의 적용:

- 처음부터 큰 자동화 프레임워크를 만들지 않는다.
- 문서, 계획, 검증, 커밋 규칙부터 작게 도입한다.
- 반복되는 작업이 생길 때만 스크립트나 하네스로 자동화한다.

## 문서별 강점

### OpenAI Harness Engineering

강점:

- 저장소를 에이전트가 읽을 수 있는 지식 시스템으로 보는 관점이 명확하다.
- `AGENTS.md`를 큰 매뉴얼이 아니라 목차로 쓰는 방향이 실용적이다.
- 테스트, 관측 가능성, UI, 로그, 메트릭을 에이전트 피드백 루프로 연결한다.
- 아키텍처 취향을 문서만이 아니라 기계적 규칙으로 강제하는 방향을 제시한다.

이 프로젝트에서의 역할:

- 전체 운영 철학과 문서 구조의 기준으로 삼는다.
- `docs/`를 기록 시스템으로 관리하는 근거로 쓴다.
- 향후 린터, 테스트, 브라우저 검증을 도입할 때 기준 문서로 삼는다.

한계:

- 구체적인 계획 문서 템플릿이나 스프린트 단위 운영 방식은 상대적으로 덜 상세하다.
- 대규모 제품 팀 경험을 바탕으로 하므로 초기 작은 저장소에는 일부가 과할 수 있다.

### OpenAI Codex 실행 계획

강점:

- 장시간 작업을 위한 계획 문서의 필수 섹션이 명확하다.
- 목적, 진행 상황, 발견 사항, 결정 로그, 검증, 복구 방법까지 포함해 중단과 재개에 강하다.
- “무엇”뿐 아니라 “왜”를 기록하라는 기준이 좋다.
- 작업 완료 조건을 관찰 가능한 동작과 명령으로 적게 한다.

이 프로젝트에서의 역할:

- `docs/plans/`의 기본 템플릿으로 삼는다.
- 복잡한 기능, 리팩터링, 마이그레이션 전에 작성할 계획 문서 기준으로 쓴다.
- 작업이 진행될수록 계획 문서도 함께 갱신하는 규칙의 근거로 쓴다.

한계:

- 평가자 분리나 브라우저 기반 QA 같은 다중 에이전트 운영 방식은 직접 다루지 않는다.
- 작은 작업에 적용하면 문서화 부담이 커질 수 있다.

### Anthropic 장시간 앱 개발 하네스 설계

강점:

- 계획자, 구현자, 평가자를 분리해 자기 평가 편향을 줄인다.
- 평가 기준을 명시적으로 만들고, 평가자가 실제 앱을 조작하게 한다.
- 스프린트 계약을 통해 높은 수준의 제품 사양을 검증 가능한 작업 단위로 바꾼다.
- 하네스 구성 요소가 실제로 필요한지 계속 검증하고 단순화하라는 관점이 좋다.

이 프로젝트에서의 역할:

- UI, 풀스택 앱, 긴 자동 구현 작업의 평가 루프 기준으로 삼는다.
- 구현자와 평가자 역할을 분리해야 할 시점을 판단하는 기준으로 쓴다.
- 스프린트 계약과 파일 기반 핸드오프를 phase/step 설계에 반영한다.

한계:

- 특정 Claude 모델과 Claude Agent SDK 실험 맥락이 포함되어 있어 그대로 이식하기 어렵다.
- 다중 에이전트 하네스는 비용과 시간이 크므로 모든 작업에 적용하면 비효율적이다.

## 이 프로젝트용 운영 모델

### 기본 작업

작은 문서 수정, 단일 파일 변경, 명확한 설정 변경은 간단히 처리한다.

운영 방식:

- 관련 파일을 읽는다.
- 최소 변경을 적용한다.
- 가능한 검증을 실행한다.
- 의미 있는 단위면 커밋한다.

### 중간 규모 작업

여러 파일을 바꾸거나 사용자가 검토해야 할 설계 판단이 있으면 짧은 계획을 둔다.

운영 방식:

- 작업 목적과 완료 기준을 먼저 적는다.
- 영향을 받는 파일과 검증 명령을 명시한다.
- 구현 후 문서와 테스트를 함께 갱신한다.
- 커밋은 논리 단위로 나눈다.

### 장시간 작업

기능 범위가 크고 중단·재개 가능성이 있거나, 여러 에이전트·리뷰 루프가 필요한 작업이다.

운영 방식:

- `docs/plans/active/`에 실행 계획을 만든다.
- 필요하면 스프린트 계약 또는 phase/step 문서를 추가한다.
- 구현자와 평가자의 관점을 분리한다.
- 진행 상황, 발견 사항, 결정 로그, 검증 결과를 계속 갱신한다.
- 완료 후 `docs/plans/completed/`로 이동하고 결과 회고를 남긴다.

## 도입 순서

1. `docs/plans/active/`와 `docs/plans/completed/` 디렉터리를 만든다.
2. OpenAI Exec Plans 기반의 한국어 실행 계획 템플릿을 만든다.
3. Anthropic 문서의 스프린트 계약을 축약한 작업 단위 템플릿을 만든다.
4. UI 작업용 브라우저 검증 체크리스트를 만든다.
5. 반복되는 phase/step 실행이 생기면 `harness_framework`를 참고해 Codex 친화 실행기를 설계한다.

## 판단 기준

작업마다 다음 질문으로 하네스 강도를 정한다.

- 이 작업은 단일 커밋으로 충분한가?
- 중단 후 다른 에이전트가 이어받아야 하는가?
- 사용자 눈에 보이는 동작이나 UI를 검증해야 하는가?
- 자기 평가가 신뢰하기 어려운 영역인가?
- 실패 시 복구 경로가 필요한가?
- 변경이 아키텍처나 장기 운영 규칙에 영향을 주는가?

대답이 많이 “예”라면 실행 계획, 평가자 분리, 브라우저 검증, phase/step 관리가 필요하다. 대부분 “아니오”라면 짧은 변경과 커밋으로 충분하다.
