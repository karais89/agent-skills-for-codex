# 초경량 하네스 선택 근거 분석

확인일: 2026-04-26

## 목적

이 문서는 초경량 하네스의 원칙을 바로 수립하기 전에, 어떤 참고 자료를 기본 골격으로 삼고 어떤 요소를 제외할지 결정하기 위한 선택 문서다. 기존의 `downloaded-repositories-analysis.md`와 `reference-documents-analysis.md`가 상세 비교 자료라면, 이 문서는 그 비교를 바탕으로 실제 도입 판단을 압축해 기록한다.

이번 단계의 산출물은 구현 계획이나 원칙 문서가 아니다. 목표는 다음 단계에서 원칙 문서를 만들 때 흔들리지 않을 기준 조합을 정하는 것이다.

## 분석 기준

| 기준 | 의미 | 높게 평가하는 조건 |
| --- | --- | --- |
| 경량성 | 초기 도입 비용과 절차 부담 | 작은 문서와 규칙만으로 시작할 수 있다. |
| Codex 적합성 | Codex 작업 방식과 권한 모델에 맞는 정도 | 저장소 문서, 계획, 검증, 커밋 흐름으로 자연스럽게 작동한다. |
| 문서 중심성 | 대화가 아니라 파일에 지식을 남기는 정도 | `AGENTS.md`, `docs/`, 계획 문서, 결정 로그로 상태를 유지한다. |
| 실행 가능성 | 바로 적용하거나 검증할 수 있는 정도 | 구체적인 명령, 체크리스트, 완료 기준으로 바꿀 수 있다. |
| 유지보수 비용 | 계속 운영할 때의 비용 | 자동화나 역할이 과하지 않고, 작은 프로젝트에서도 갱신 가능하다. |

## 분류 기준

| 분류 | 의미 |
| --- | --- |
| 채택 | 초경량 하네스의 기본 골격으로 삼는다. |
| 부분 채택 | 특정 원칙, 문서 형식, 검증 방식만 가져온다. |
| 보류 | 가치가 있지만 현재 단계에서는 비용이 커서 나중에 재검토한다. |
| 제외 | 초기 하네스 범위에서는 사용하지 않는다. 참고 자료로만 둔다. |

## 저장소 선택 분석

### 전체 판단

| 저장소 | 분류 | 핵심 판단 |
| --- | --- | --- |
| `agent-skills` | 채택 | 생명주기 구조가 가볍고 Codex 작업 흐름으로 번역하기 쉽다. |
| `andrej-karpathy-skills` | 채택 | 모든 작업에 적용할 기본 행동 원칙으로 가장 작고 선명하다. |
| `openai-codex` | 부분 채택 | 실제 제품 수준의 기준점으로 유용하지만 구조를 직접 모방하기에는 무겁다. |
| `superpowers` | 부분 채택 | 계획, TDD, 리뷰 규율은 강력하지만 전체 워크플로우는 초기에 과하다. |
| `gstack` | 부분 채택 | 역할 분리와 UI 검증 기준은 유용하지만 명령과 역할 수가 많다. |
| `harness_framework` | 보류 | phase/step 실행 구조는 유용하지만 현재는 Codex 이식 비용이 크다. |

### `agent-skills`

분류: 채택

선택 근거:

- `define → plan → build → verify → review → ship` 흐름이 초경량 하네스의 기본 생명주기로 적합하다.
- 스킬, 페르소나, 명령의 역할 구분이 명확해 향후 문서 구조로 확장하기 쉽다.
- `AGENTS.md`를 짧은 진입점으로 두고 세부 지식을 별도 파일에 분리하는 방향과 맞다.
- 컨텍스트 엔지니어링, 원자적 커밋, 검증 루프 같은 규칙이 현재 프로젝트 지침과 충돌하지 않는다.

채택할 것:

- 작업 생명주기 명칭과 순서
- 짧은 진입점과 세부 문서 분리
- 스펙, 계획, 구현, 검증, 리뷰, 배포의 단계적 사고
- 커밋을 의미 있는 작업 단위의 저장점으로 사용하는 방식

제외할 것:

- 모든 스킬을 한 번에 도입하는 방식
- 아직 실제 반복 업무가 없는 단계에서의 광범위한 명령 체계

판단:

초경량 하네스의 기본 골격은 `agent-skills`식 생명주기를 따른다. 다만 초기 문서는 최소 세트로 시작하고, 반복 작업이 확인될 때만 스킬이나 명령을 늘린다.

### `andrej-karpathy-skills`

분류: 채택

선택 근거:

- 원칙 수가 적고, 거의 모든 코딩 작업에 적용할 수 있다.
- 과한 추상화, 확인 없는 가정, 주변 코드 변경, 검증 없는 완료 선언을 직접 억제한다.
- 별도 도구나 실행기 없이 문서 규칙만으로 적용 가능하다.

채택할 것:

- 먼저 생각하고 작게 바꾸는 원칙
- 사용자 요청과 직접 연결되지 않는 변경을 피하는 기준
- 성공 기준을 먼저 세우고 검증 후 완료하는 루프
- 단순성을 기본값으로 두는 태도

제외할 것:

- 모든 작업에 동일한 강도의 숙고를 요구하는 운영 방식
- 프로젝트별 검증 체계 없이 일반 행동 지침만으로 완료를 판단하는 방식

판단:

`andrej-karpathy-skills`는 초경량 하네스의 행동 규범으로 채택한다. 다만 실행 계획, 문서 색인, 검증 명령 같은 운영 구조는 다른 자료로 보완한다.

### `openai-codex`

분류: 부분 채택

선택 근거:

- 실제 Codex 제품 저장소로서 에이전트 지침, 테스트, 리뷰, 모듈 분리 기준을 확인할 수 있다.
- 큰 저장소에서 `AGENTS.md`와 스킬, 플러그인, 테스트 규칙을 어떻게 운영하는지 보여준다.
- Codex와 직접 관련된 저장소라 용어와 작업 모델의 참고 가치가 높다.

채택할 것:

- 구체적인 작업 지침을 저장소 안에 두는 방식
- 리뷰 품질을 변경 크기, 테스트, 호환성, 사용자 영향으로 나누는 관점
- 큰 공통 모듈을 피하고 새 개념을 적절히 분리하는 원칙
- 사용자에게 보이는 변화에는 스냅샷이나 화면 검증을 요구하는 태도

제외할 것:

- Rust 워크스페이스, Bazel, 대규모 크레이트 구조 같은 제품 구현 세부
- 플러그인과 앱 서버를 포함한 전체 아키텍처 모방
- 초기 단계부터 대규모 테스트·리뷰 체계를 전부 도입하는 방식

판단:

`openai-codex`는 직접 따라 할 템플릿이 아니라 상한선이자 품질 기준점으로 둔다. 초경량 하네스에는 제품 구조가 아니라 지침의 구체성, 검증 문화, 모듈화 태도만 반영한다.

### `superpowers`

분류: 부분 채택

선택 근거:

- 계획 작성, TDD, 코드 리뷰 요청, 브랜치 마무리까지 강한 개발 규율을 갖춘다.
- 작업 단위를 작게 쪼개고 각 단계에 기대 결과와 검증 명령을 적는 방식이 실행 계획과 잘 맞다.
- 스펙 준수 리뷰와 코드 품질 리뷰를 분리하는 관점이 유용하다.

채택할 것:

- 복잡한 작업에서 계획을 먼저 쓰고 체크박스로 갱신하는 방식
- 검증 전 완료 금지
- 리뷰를 스펙 준수와 코드 품질로 나누는 관점
- 스킬 자체를 행동 개선 도구로 평가하는 기준

제외할 것:

- 모든 단계에 상세 계획, 테스트 코드, 커밋 명령까지 요구하는 고강도 절차
- 작은 변경까지 워크트리와 다중 리뷰 흐름으로 확장하는 방식

판단:

`superpowers`는 초경량 하네스의 기본값이 아니라 복잡한 작업에서 강도를 높일 때 참고한다. 특히 계획과 검증 규율만 축약해 가져온다.

### `gstack`

분류: 부분 채택

선택 근거:

- 제품, 엔지니어링, 디자인, QA, 보안, 릴리스 역할을 분리하는 방법이 풍부하다.
- UI 작업에서 브라우저 조작, 스크린샷, 실제 사용자 흐름 검증을 강조한다.
- 자동 계획과 다중 리뷰 흐름은 장시간 작업에서 품질을 높일 수 있다.

채택할 것:

- UI 작업에서 실제 브라우저 검증을 요구하는 기준
- 제품 검토, 설계 검토, QA 검토를 구분하는 관점
- 릴리스 후 문서 갱신을 별도 단계로 보는 태도
- 위험 명령, 파일 범위, 자동화 범위를 제한하는 안전 개념

제외할 것:

- 많은 slash command와 역할을 초기부터 모두 도입하는 방식
- 개인 환경, 텔레메트리, 브라우저 서버, 자동 업데이트에 결합된 운영 구조
- 초경량 문서 하네스를 넘어서는 제품 조직 시뮬레이션

판단:

`gstack`은 프론트엔드와 릴리스 품질을 높이는 참고 자료로 둔다. 초기 원칙에는 역할 이름을 많이 도입하지 않고, 검증 기준과 안전 개념만 가져온다.

### `harness_framework`

분류: 보류

선택 근거:

- 한국어 문서 중심, phase/step 구조, 상태 파일, 재시도, 커밋 분리 같은 요소는 현재 프로젝트와 방향이 맞다.
- 장시간 작업을 독립 세션으로 나누고 step summary를 다음 단계에 넘기는 방식은 컨텍스트 관리에 유용하다.
- 다만 현재 구현은 Claude CLI와 설정에 강하게 결합되어 있다.

채택할 수 있는 것:

- `phases/{task}/index.json`과 `stepN.md`처럼 작업을 기계적으로 나누는 구조
- 각 step에 목표, 입력 파일, 금지사항, 검증 기준을 포함하는 원칙
- 완료 summary를 다음 step의 컨텍스트로 넘기는 방식

보류할 것:

- 실행 스크립트의 직접 이식
- Claude CLI 호출부와 hook 설정
- 넓은 `git add -A` 기반 자동 스테이징
- 상태 전이와 재시도 자동화 전체

판단:

`harness_framework`는 초경량 하네스 v0.0에는 넣지 않는다. 먼저 문서, 계획, 검증, 커밋 규칙으로 충분한지 확인하고, 반복되는 장시간 작업이 생기면 Codex 권한 모델에 맞춰 별도 설계를 검토한다.

| OpenAI Codex 실행 계획 | 채택 | 큰 작업을 살아 있는 계획 문서로 관리하는 방식이 Codex에 직접 적합하다. |

### OpenAI Harness Engineering

분류: 채택

선택 근거:

- `AGENTS.md`를 짧은 진입점으로 유지하고, 깊은 지식은 `docs/`에 두는 현재 프로젝트 지침과 일치한다.
- 에이전트가 읽을 수 없는 지식은 운영 지식이 아니라는 관점이 초경량 하네스의 핵심 전제다.
- 테스트, 린터, 로그, UI 확인 같은 피드백 루프를 하네스의 일부로 보는 점이 실용적이다.

채택할 것:

- 저장소 중심 지식 관리
- 문서, 테스트, 검증 명령을 에이전트 피드백 루프로 설계하는 관점
- 아키텍처 취향을 문서와 기계적 규칙으로 함께 남기는 방식

제외할 것:

- 큰 조직이나 대형 제품 운영을 전제로 한 모든 체계
- 아직 필요하지 않은 관측 가능성, 메트릭, 복잡한 CI 설계

판단:

초경량 하네스의 가장 상위 철학은 OpenAI Harness Engineering을 따른다. 즉, 하네스는 별도 거대 프레임워크가 아니라 에이전트가 저장소 안에서 읽고 실행하고 검증할 수 있게 만드는 환경이다.

### OpenAI Codex 실행 계획

분류: 채택

선택 근거:

- 복잡한 기능, 큰 리팩터링, 장시간 작업 전에 실행 계획을 작성하거나 갱신한다는 프로젝트 지침과 직접 연결된다.
- 계획을 일회성 할 일 목록이 아니라 진행 상황, 발견, 결정, 검증 결과가 쌓이는 살아 있는 문서로 본다.
- Codex가 중단 후 재개하거나 다른 에이전트가 이어받을 때 필요한 정보를 명확히 제공한다.

채택할 것:

- 목적, 진행 상황, 발견 사항, 결정 로그, 검증 기준, 복구 방법을 포함하는 계획 문서 구조
- 작업 중 계획 문서를 계속 갱신하는 규칙
- 완료 조건을 관찰 가능한 동작과 명령으로 쓰는 방식

제외할 것:

- 작은 문서 수정이나 단일 파일 변경에까지 전체 계획 문서를 요구하는 방식
- 계획 문서를 작성했지만 실제 변경 중 갱신하지 않는 형식적 운영

판단:

복잡한 작업의 계획 형식은 OpenAI Codex 실행 계획을 기본으로 한다. 단, 초경량 하네스에서는 작업 규모에 따라 계획 문서의 깊이를 조절한다.

### Anthropic Harness Design

분류: 부분 채택

선택 근거:

- 계획자, 구현자, 평가자를 분리해 자기 평가 편향을 줄이는 구조가 명확하다.
- UI와 앱 품질을 실제 브라우저 조작과 평가 루프로 검증하는 방식이 강력하다.
- 스프린트 계약과 파일 기반 핸드오프는 장시간 작업의 컨텍스트 손실을 줄인다.

채택할 것:

- 중요한 작업에서 구현 관점과 평가 관점을 분리하는 원칙
- UI 작업의 브라우저 조작, 스크린샷, 사용자 흐름 검증
- 작업 단위별 완료 기준을 먼저 합의하는 스프린트 계약 개념
- 하네스 구성 요소를 계속 단순화하거나 제거하며 비용을 점검하는 태도

제외할 것:

- 모든 작업에 다중 에이전트 구조를 적용하는 방식
- 특정 Claude SDK나 모델 실험 맥락에 묶인 구현 방식
- 제품 사양 확장, 구현, 평가를 항상 별도 단계로 강제하는 운영

판단:

Anthropic 문서는 초경량 하네스의 기본 골격이 아니라 강화 옵션이다. UI, 긴 자동 구현, 자기 평가가 약한 영역에서 평가자 분리와 브라우저 검증을 도입한다.

## 결론

초경량 하네스의 기본 후보는 `OpenAI Harness Engineering + agent-skills`가 맞다. 여기에 `andrej-karpathy-skills`를 행동 규범으로 결합하고, 큰 작업에는 OpenAI Codex 실행 계획을 붙인다. `superpowers`, `gstack`, Anthropic Harness Design은 하네스 강도를 높이는 선택적 보강재로 둔다. `harness_framework`는 방향은 맞지만 v0.0에 바로 넣기에는 실행기 이식과 유지보수 비용이 커서 보류한다.
