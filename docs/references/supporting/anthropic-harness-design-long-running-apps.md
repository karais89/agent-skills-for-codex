# Anthropic 장시간 앱 개발 하네스 설계 참고

원문: https://www.anthropic.com/engineering/harness-design-long-running-apps

확인일: 2026-04-26

## 핵심 요약

Anthropic의 이 문서는 장시간 자율 앱 개발에서 모델 성능을 끌어올리기 위한 하네스 설계를 설명한다. 단일 에이전트에게 긴 작업을 끝까지 맡기기보다, 계획 수립자, 구현자, 평가자를 분리하고 각 단계 사이에 명확한 산출물과 피드백 루프를 둔다. 특히 주관적인 프론트엔드 품질과 검증 가능한 앱 동작 모두에서 평가 기준을 구체화하고, 구현 에이전트와 평가 에이전트를 분리하는 것이 핵심이다.

## 주요 개념

- 긴 작업에서는 컨텍스트가 차거나 작업이 길어질수록 일관성이 떨어질 수 있다.
- 컨텍스트 리셋은 새 에이전트가 구조화된 핸드오프 문서를 받아 이어서 작업하게 하는 방식이다.
- 컴팩션은 대화를 요약해 같은 에이전트가 계속 진행하게 하는 방식이며, 리셋과 장단점이 다르다.
- 자기 평가만으로는 품질을 과대평가하기 쉽기 때문에 작업자와 평가자를 분리한다.
- 평가자는 기본적으로 관대해지기 쉬우므로, 구체적인 기준과 예시로 비판적으로 조정해야 한다.
- 프론트엔드 품질은 디자인 품질, 독창성, 기술적 완성도, 기능성 같은 기준으로 나누어 평가할 수 있다.
- 평가자는 정적 스크린샷만 보지 않고 Playwright 같은 도구로 실제 앱을 조작하며 판단하는 편이 좋다.

## 세 에이전트 구조

- 계획 수립자: 짧은 사용자 요청을 제품 사양으로 확장한다. 세부 구현을 과도하게 지정하기보다 산출물, 사용자 가치, 높은 수준의 기술 방향을 정한다.
- 구현자: 사양을 바탕으로 기능을 구현한다. 큰 작업은 스프린트나 명확한 작업 단위로 나누고, 각 단위가 끝날 때 자체 점검을 수행한다.
- 평가자: 실행 중인 앱을 직접 조작하면서 UI, API, 데이터 상태, 기능 완성도, 코드 품질을 검증한다. 기준 미달 항목이 있으면 구체적인 피드백을 남기고 구현자가 다시 반복하게 한다.

## 스프린트 계약

스프린트 계약은 구현 전에 구현자와 평가자가 해당 작업 단위의 완료 기준을 합의하는 문서다. 제품 사양이 의도적으로 높은 수준에 머무를 때, 스프린트 계약은 사용자 스토리와 검증 가능한 구현 사이의 간극을 줄인다.

계약에는 다음 내용을 포함한다.

- 이번 작업 단위에서 만들 기능
- 완료로 인정할 사용자 관찰 가능 동작
- 평가자가 실행할 테스트와 확인 절차
- 제외할 범위와 다음 작업으로 넘길 항목

## 파일 기반 핸드오프

문서에서는 에이전트 간 커뮤니케이션을 파일로 처리했다. 한 에이전트가 파일을 작성하면 다른 에이전트가 읽고 같은 파일에 응답하거나 새 파일을 작성한다. 이 방식은 대화 맥락에만 의존하지 않고, 현재 상태와 합의 내용을 저장소 안에 남긴다는 점에서 장시간 작업에 유리하다.

## 하네스 단순화 원칙

하네스의 각 구성 요소는 현재 모델이 혼자 잘하지 못한다고 가정하는 부분을 보완한다. 모델 성능이 향상되면 기존 하네스의 일부가 불필요한 비용과 지연이 될 수 있다. 따라서 한 번에 구조를 크게 바꾸기보다, 구성 요소를 하나씩 제거하거나 바꾸면서 어떤 부분이 실제로 품질에 기여하는지 확인해야 한다.

## 프로젝트에 적용할 원칙

- 복잡한 앱이나 도구를 만들 때는 실행 계획과 평가 기준을 먼저 명확히 한다.
- 작업자와 평가자의 역할을 분리해 자기 평가 편향을 줄인다.
- UI 작업은 실제 브라우저 조작, 스크린샷, 사용자 흐름 테스트를 검증 루프에 포함한다.
- 큰 기능은 스프린트 계약처럼 완료 기준이 합의된 작은 단위로 나눈다.
- 장시간 작업의 중간 상태는 대화가 아니라 저장소의 문서와 파일에 남긴다.
- 하네스와 자동화는 고정된 정답으로 보지 않고, 현재 모델과 작업 난이도에 맞게 계속 단순화하거나 강화한다.
